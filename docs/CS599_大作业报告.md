## 一、选题背景与设计思想

### 1.1 问题定义

烹饪是日常生活的高频场景，但厨房新手面临三大痛点：

1. **知识获取困难**：想做一道菜，需要跨多个平台搜索做法，信息碎片化严重
2. **缺乏个性化指导**：网上菜谱不考虑用户技能水平——新手和老手看到的是同一份教程
3. **安全隐患**：传统菜谱网站缺乏安全提示，新手容易忽略食品安全风险

### 1.2 现有方案不足

| 现有方案 | 不足 |
|---------|------|
| 菜谱 App（下厨房、豆果美食） | 静态内容，不支持自然语言问答，无个性化 |
| 通用 AI 助手（ChatGPT） | 可能编造菜谱（幻觉），无结构化知识库约束 |
| 搜索引擎 | 信息碎片化，需要用户自行筛选整合 |

### 1.3 项目价值

KitchenPilot 是一个**垂直领域 Agentic AI 系统**，核心价值：

- **RAG 约束**：所有回答基于结构化菜谱知识库，消除 LLM 幻觉
- **多 Agent 协作**：Supervisor 编排菜谱问答、推荐、安全检查三个专职 Worker 并行工作
- **个性化**：根据用户技能画像（新手/入门/老手）调整回答风格和推荐策略
- **安全第一**：LLM 审查 + 规则引擎双重安全检查

### 1.4 技术路线

采用 **SDD（规格驱动开发）** 方法论，先定义 Product Spec → Architecture Spec → API Spec，再进入实现阶段。技术选型以 LangGraph 多 Agent 架构为核心，DeepSeek API 为 LLM 底座，BM25 为检索引擎，SSE 实现 Token 级流式输出。

## 二、Specs 规格文档

### 2.1 Product Spec（产品规格）

#### 用户故事

| ID | 用户 | 需求 | 验收标准 |
|----|------|------|---------|
| US-01 | 厨房新手 | 询问某道菜的做法 | 输入菜名，返回步骤列表 + 新手提示 + 安全提醒 |
| US-02 | 厨房新手 | 根据现有食材推荐菜谱 | 输入食材列表，返回 Top-3 推荐 + 匹配/缺少食材说明 |
| US-03 | 厨房新手 | 询问失败原因 | 输入"XX菜为什么失败"，返回失败排查指南 |
| US-04 | 所有用户 | 多轮对话追问 | 问完做法后追问"放多少盐"，系统理解上下文 |
| US-05 | 所有用户 | 切换用户画像 | 切换新手/老手后，推荐结果随之变化 |

#### 功能边界

- **包含**：菜谱做法问答、食材推荐、失败排查、替代方案查询、个性化推荐
- **不包含**：视频教程、营养计算、购物清单生成、社区互动

### 2.2 Architecture Spec（架构规格）

#### Agent 架构决策

| 决策 | 选型 | 理由 |
|------|------|------|
| Agent 模式 | Supervisor + Multi-Worker | 菜谱问答和食材推荐是独立任务，适合并行分发 |
| 检索方案 | BM25（中文 bigram） | 无需外部嵌入服务，本地零依赖，中文效果足够 |
| 记忆方案 | MemorySaver（thread_id） | LangGraph 内置，自动拼接多轮历史 |
| 流式方案 | httpx 直连 DeepSeek SSE | 绕过 LangChain 封装层，实现真正的 Token 级流式 |
| 质检方案 | 规则引擎（确定性） | 食品安全检查不需要 LLM，确定性规则更可靠 |

#### Worker 职责划分

| Worker | 绑定工具 | 触发条件 |
|--------|---------|---------|
| recipe_qa_worker | `search_recipes` | 用户问做法/技巧/失败原因/替代方案 |
| recommend_worker | `recommend_by_ingredients` + `get_user_profile` | 用户提供食材或要求推荐 |
| safety_worker | 无（纯 LLM 审查） | 回答生成后，检查安全隐患 |

### 3.2 Agent 交互流程

```
1. 用户发送 "番茄炒蛋怎么做？推荐类似菜"
2. Supervisor (analyze 节点) 分析：
   → "涉及做法 → recipe_qa_worker"
   → "涉及推荐 → recommend_worker"
3. Send API 并行分发：
   - recipe_qa_worker: chat → search_recipes("番茄炒蛋怎么做") → chat
   - recommend_worker: chat → get_user_profile("demo_user") → recommend_by_ingredients("鸡蛋,番茄") → chat
4. 两个 Worker 同时返回结果
5. Supervisor 再次 analyze → "DONE"
6. aggregate 节点汇总 + 规则引擎质检
7. 最终回答通过 httpx stream → SSE token 级推送到前端
```

### 3.3 数据流设计

```
用户 query → [当前用户ID: xxx]\n query  ← 注入 user_id
  → HumanMessage → AgentState.messages
  → Supervisor 分析 → Worker 消息追加到 state
  → 工具调用 → ToolMessage 追加到 state
  → 汇总 → 提取 tool_outputs → 拼接 context
  → httpx.stream(DeepSeek API) → token → SSE → 前端逐字渲染
```

### 4.1 Agent 核心循环

Supervisor 图定义（[supervisor_graph.py](../src/supervisor_graph.py)）：

```python
# analyze 节点：Supervisor LLM 分析需求
def analyze_node(state: SupervisorState) -> dict:
    count = state.get("analyze_count", 0)
    if count >= 4:
        return {"messages": [AIMessage(content="DONE")], "analyze_count": 0}
    result = (analysis_prompt | model).invoke({"messages": state["messages"]})
    return {"messages": [result], "analyze_count": count + 1}

# 路由：并行分发到多个 Worker
def route_after_analyze(state: SupervisorState):
    content = str(state["messages"][-1].content).strip().lower()
    if "done" in content:
        return "aggregate"
    return [Send(node=name, arg={"messages": state["messages"]})
            for name in selected_workers]
```

### 4.5 检索引擎

[retriever.py](../src/rag/retriever.py) — 纯 BM25 + 规则重排序：

```python
class RecipeRetriever:
    def search(self, query, top_k=4):
        bm25_results = self._bm25.search(query, max(top_k * 2, 8))
        return self._rerank(query, bm25_results)[:top_k]

    def _rerank(self, query, chunks):
        # 分块类型偏好加权 + 实体（菜名/食材）匹配大幅加分
        boost = priority.get(c.chunk_type, 0) * 4 + self._entity_boost(query, c)
        return sorted(chunks, key=..., reverse=True)
```

## 六、系统升级与扩展

### 6.1 可扩展架构

当前 Supervisor + Worker 架构天然支持扩展：

- **新增 Worker**：只需实现 `create_worker(tools, prompt, name)`，在 Supervisor 提示词中注册即可
- **新增工具**：`@tool` 装饰器一键添加，ToolNode 自动识别
- **替换 LLM**：改 `.env` 中 `DEEPSEEK_BASE_URL` 即可切换到任何 OpenAI 兼容接口

### 6.2 下一阶段计划

| 优先级 | 计划 | 说明 |
|--------|------|------|
| P0 | 持久化会话存储 | 当前 MemorySaver 重启丢失，替换为 SQLite/Redis |
| P1 | MCP 协议集成 | 将菜谱检索封装为 MCP Server，Agent 通过 MCP Client 调用（+3 加分项） |

## 七、课程总结

### 7.1 个人收获

1. **从"写代码"到"编排 Agent"**：理解了 LangGraph StateGraph 的节点、条件边、Send 并行分发等核心概念，能够设计 Supervisor + Worker 的多 Agent 协作架构。

2. **SDD 方法论实践**：先写 Spec 再写代码，Product Spec → Architecture Spec → API Spec 的流程让实现有据可依，避免了边写边改的混乱。

3. **中文 RAG 的工程挑战**：BM25 bigram 分词、重排序、实体加权——这些看似简单的技术组合起来才能在实际场景中达到可用水平。

4. **SSE 流式的细节**：从 `EventSourceResponse` 缓冲问题到 `httpx` 直连 DeepSeek stream API，理解了流式输出的完整链路。

### 7.2 工程思维转变

- **从"调 API"到"设计系统"**：不再把 LLM 当作黑箱调用，而是作为 Agent 系统中的一个组件来编排
- **容错与降级**：如果 Worker 失败、LLM 超时、检索为空，每一层都需要 graceful degradation
- **安全不可外包给 LLM**：确定性规则引擎 + 否定语境检测，比"让 LLM 自己注意安全"可靠得多

### 7.3 对课程的建议

1. 建议增加 **MCP 协议实战环节**——当前课程提到 MCP 但缺乏动手指导
2. 建议提供 **统一的评估基准**——不同项目的 Agent 评估标准差异大，统一基准有助于横向对比
3. 课程时间偏紧，建议 Proposal 阶段提前到第 11-12 周，给实现留更多时间
